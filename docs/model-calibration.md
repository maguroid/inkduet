# モデル接続時の較正ログ

## 目的

実モデルへ接続したときにだけ分かる出力変動、構造化出力、待ち時間、失敗・再試行、UXへの影響を
一次観測として残し、設計判断、実装Issue、新しい課題仮説へ漏れなく移す。

この文書は進行状態や完成条件の正本ではない。実装の進行はM IssueとDev Board、課題・機能仮説は
BL Issue、採用した設計は各設計文書を正本とする。

## 実行タイミング

- M0b: 仮UIから実モデルへ接続する薄いウォーキングスケルトン
- M1〜M4: 各縦断Issueの実モデル評価ゲート
- M5: 工程別の観測を統合した専用UI対チャットの価値評価

M5を最初の実モデル評価にしない。M0bで基盤上の制約を発見し、M1〜M4で工程固有の問題を継続的に
回収する。

## 毎回固定・記録する条件

- 対象Issueとアプリのコミット
- モデル名、モデル設定、Codexバージョン
- `article-writing-core` Skillと出力スキーマの版
- 入力シナリオと期待する結果
- 最初のイベントまでの時間、完了までの時間、再試行回数
- 実行結果と人間の評価

入力本文に非公開情報が含まれる場合は、この文書へ全文を複製せず、再現に必要な特徴だけを記録する。

## 観測の分類

- `quality`: 主張、ストーリー、文章、修正案、影響範囲の品質
- `schema`: 欠落、型違反、過剰出力、途中終了
- `latency`: 初回表示、完了までの待ち時間、ストリーミングの見え方
- `recovery`: キャンセル、再試行、部分失敗、状態復元
- `ux`: 比較、選択、承認、却下、確認のしやすさ
- `context`: 対象範囲や前提の取り違え、版やストーリーとの対応崩れ

## 観測後の移送先

1. まず本書へ、期待・実際・影響を事実として記録する。
2. UX、状態モデル、入出力契約の判断が変わったら、該当設計文書へBefore / After / 理由を追記する。
3. 既存の縦断体験で直す実装作業は、該当するM Issueへ追加する。
4. 新しく見つかった解くべき課題や機能仮説は、`BL-*` Issueとして作成する。
5. すぐ直さない場合も、許容する理由または後続Issueを記録し、観測だけを放置しない。

## 記録テンプレート

```markdown
### CAL-YYYYMMDD-NN: 短い観測名

- 対象Issue:
- アプリのコミット:
- モデル / 設定 / Codex:
- Skill / 出力スキーマ:
- 入力シナリオ:
- 期待:
- 実際:
- 時間: 初回イベント / 完了
- 再試行回数:
- 分類: quality / schema / latency / recovery / ux / context
- 人間への影響:
- 判断:
- 移送先: 設計文書 / M Issue / BL Issue / 許容
```
