# App Serverの技術設計

## 目的

Codexの開発向け標準体験をそのまま流用せず、記事制作の判断順序へ最適化した専用クライアントを作る。
Codex App Serverが提供するスレッド、ターン、承認、ストリーミングイベントを基盤とし、記事固有の
進行状態と成果物はアプリ側で明示的に管理する。

## Skillだけで完結させない理由

記事制作の全工程はCodex Skillでも実行できるため、「Skillでは不可能」は専用アプリの根拠にしない。
専用アプリは、中心主張、ストーリー、本文、レビュー、変更差分、確認状態を同時に表示し、対象選択、
案比較、部分採用、差分承認を直接操作できるようにする。

検証すべき問いは、記事を生成できるかではなく、人間の比較・選択・承認をチャットより少ない負担で
行えるかである。Markdown、Git差分、会話上の要約だけで十分なら、専用アプリは不要でありSkillだけを残す。

## UI/UXプロトタイプを技術基盤より先に作る

App Server接続や永続化へ着手する前に、一本のサンプル記事データで「伝えたいこと」から完成宣言まで
操作できるクリック可能なプロトタイプを作る。先に確定するのは、画面構成、情報の同時表示、制作フェーズ、
人間の承認点、前工程へ戻る挙動、対象選択から差分確認までの中心編集ループである。

仮UIは、背景、文字、罫線、アイコン、図形、装飾など、状態を示さない要素をグレースケールにする。
有彩色は、選択、現在工程、処理中、承認、未確認、警告、エラーなど、明示的な状態の表示だけに使う。
状態は色だけで伝えず、ラベル、アイコン、形、位置のいずれかを併用する。状態色の具体的な色相、
ブランド表現、細かな余白やアニメーションは、この段階では固定しない。

記事状態モデル、App Serverへの入出力、コンポーネント境界は、採用したUXの操作と状態から逆算する。
技術基盤を先に作り、その都合へUIを合わせない。プロトタイプは完成デザインではなく、M1以降の実装で
操作上の問題が見つかれば更新する操作契約として扱う。

## 実モデルで継続的に較正する

M0aはモックデータで操作の理解しやすさを検証する。一方、出力の長さ・構造・品質のばらつき、
ストリーミング、待ち時間、キャンセル、再試行、構造化出力の欠落は実モデルへ接続しなければ分からない。

M0bでは、完成画面を作る前に仮UIから実モデルへ接続し、ストーリー案と3つの修正案を生成する薄い
ウォーキングスケルトンを通す。ここで状態モデル、出力スキーマ、失敗時の復旧経路を早期に較正する。
M1〜M4でも各縦断体験の完了条件として実モデルを複数ケースで実行し、工程ごとの問題を回収する。
M5は初めて実モデルを試す場ではなく、較正済みの工程を統合して専用UIの価値を評価する場とする。

観測は[モデル接続時の較正ログ](./model-calibration.md)へ記録する。UXや状態モデルの判断が変わったら
該当設計文書へBefore / After / 理由を残し、実装作業は該当M Issue、新しい課題仮説はBL Issueへ移す。

## 責務分担

| 層 | 所有するもの |
| --- | --- |
| `article-writing-core` Skill | 読者設定、ストーリー設計、調査、執筆、レビュー、修正、完成判定の原則・参考資料・テンプレート |
| 専用アプリ | 制作フェーズ、記事データ、状態遷移、人間の承認、版・差分、採否、確認状態、評価指標、UI |
| `developerInstructions` | 指定フェーズだけを処理する、人間の承認を推測しない、アプリの状態を正本とする等の実行契約 |
| `outputSchema` | 各フェーズからアプリへ返す構造化データの機械的な契約 |

Skill自身はフェーズを進めず、アプリも記事制作の判断基準を重複して持たない。
MVPでは工程ごとにSkillを分割せず、一つの`article-writing-core` Skillからフェーズ別の参考資料を読む。
独立利用の需要が確認できた工程だけを後から別Skillへ切り出す。

## App Serverからの明示的なSkill呼び出し

Skillは本リポジトリへ同梱してバージョンを固定する。App Serverクライアントは、
`skills/extraRoots/set`と`skills/list`で対象を解決し、`turn/start`の入力へ`type: "skill"`、
名前、パスを渡して明示的に発動する。

これにより、利用者の普段使い環境へ偶然インストールされているSkillや、プロンプトによる
暗黙マッチングへ依存しない。

## 実行環境の分離

`codex app-server`は、明示的に分離しなければ利用者のCodex環境を継承しうる。
ハーネス専用の`CODEX_HOME`を用意し、認証と必要最小限の設定だけを管理する。

`thread/start`では作業ディレクトリを絶対パスで渡し、`config.project_doc_max_bytes`を`0`にする。
起動時には次を検証する。

- `initialize.result.codexHome`が専用パスと一致する。
- `thread/start.result.instructionSources`が空である。
- 同梱した`article-writing-core` Skillだけを列挙・明示発動できる。

`project_doc_max_bytes = 0`は起動時のプロジェクト指示の自動注入を止める設定であり、読み取り可能な
ファイルをセキュリティ境界として隔離するものではない。未信頼の作業フォルダを扱う場合は、専用の
コンテナ、仮想ファイルシステム、または指示ファイルを除外した作業コピーも検討する。

## 実装準備の完了条件

- サンプルデータによる全体UXプロトタイプで、制作フローと中心編集ループを一巡できる。
- 採用した画面構成、操作、イベント、状態遷移、人間の承認点、未解決事項を記録する。
- 状態以外のUI要素をグレースケールにし、有彩色を状態表示だけに限定する。
- グレースケール表示でも、ラベル、アイコン、形、位置から状態を判別できる。
- 仮UIから実モデルへ接続し、ストーリー案と3つの修正案を構造化出力で受け取る。
- ストリーミング、待ち時間、キャンセル、再試行、構造化出力の逸脱を観測して較正ログへ記録する。
- 代表ユーザー、代表シナリオ、一本のサンプル入力を決める。
- ワークフローの状態遷移、各段階の入出力、人間が承認する箇所を定義する。
- 安定面の`stdio`で`initialize`、`thread/start`、`turn/start`、イベント表示まで接続する。
- 固定したCodexバージョンからTypeScriptスキーマを生成する。
- App Serverのスレッド履歴と、記事側の永続データの責務分担を決める。
- `article-writing-core` Skillの原則、フェーズ別参考資料、アプリとの入出力契約を決める。
- App Serverから同梱Skillを列挙・明示発動できることを確認する。
- MVP、後回し範囲、受け入れ条件、実装順序を文書とGitHub Issueへ落とす。
